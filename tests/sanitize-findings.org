#+title: Libregnum sanitizer leak findings (LSan triage)
#+options: toc:t

This is the triage of the library-internal leaks ("D" items) surfaced by
=make test-sanitize= after the test-code leaks (B), the crash-reporter
singleton (C), and the =test-scripting*= runtime noise (A) were resolved /
excluded. It exists to give each finding a home with a stack trace until the
underlying library bug is fixed; each item is also filed as a vimban ticket.

* How these were produced

#+begin_src sh
make test-sanitize        # builds into build/sanitize/, runs suite under ASan+UBSan+LSan
#+end_src

The run sets =G_SLICE=always-malloc= + =G_DEBUG=gc-friendly= so GLib's slice
allocator does not hide allocations, =ASAN_OPTIONS=detect_leaks=1=, and
=LSAN_OPTIONS=suppressions=tests/lsan.supp= (the crash-reporter singleton is
suppressed there). =test-scripting*= are excluded (see =SANITIZE_BINARIES= in
=tests/Makefile=). 0 ASan/UBSan memory-safety errors and 0 assertion failures
across the suite; the only remaining failures are the direct leaks below.

Each "root-cause frame" is the deepest application frame in the leak's stack
-- the libregnum =src/= caller that triggered the allocation (yaml-glib and
graylib frames above it are dependencies allocated on libregnum's behalf).
Extracted with an awk pass over the LSan output grouping leak blocks by
binary + caller frame.

* Common root cause

The dominant theme is **serialization paths leak the YAML document/builder
they construct.** =*_save_to_file= / =*_save_to_data= routines build a
=YamlDocument= / =YamlBuilder= (or =LrgSaveContext= string tree) and return
without unreffing the intermediate object(s). yaml-glib allocates on
libregnum's behalf, so the leak is attributed to the libregnum caller.

This is a single class of bug repeated across modules: input-map, save,
scene, atlas, tutorial, editor, and (via the run/level object graphs)
deckbuilder. Fixing the serialization helpers to =g_object_unref= / free
their temporaries on every return path will clear the bulk of these.

The two =test-template-*= items are different: 56-byte =datalist_append=
(qdata) allocations that leak during =g_object_unref= of the *test's* mock
objects (e.g. =TestDailyRewardsMock= at
=tests/test-template-engagement.c:44=). These originate in the test mock's
teardown, not the library, though they reflect how the engine's engagement /
idle systems attach qdata to the mock. They are filed together as one
test-mock ticket.
